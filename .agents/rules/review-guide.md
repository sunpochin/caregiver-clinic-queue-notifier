# 🔍 AI 審查規範指引 (Review Guide)

本文件定義本專案（高醫看診進度通知器）的程式碼審查標準。當 AI 助理協助審查、重構或編寫程式碼時，請務必遵循以下規範：

---

## 1. ✍️ 註解撰寫規範 (Comment Standards)

> [!IMPORTANT]
> **雙軌並行註解**：本專案的所有程式碼註解必須同時包含 `[技術]` 與 `[小朋友]` 兩個版本，以並行或上下堆疊的方式呈現：
> * **`[技術]`**：以簡明、專業的軟體工程語言解釋該段程式碼的架構設計、演算法邏輯或特定防呆機制。
> * **`[小朋友]`**：使用生動的日常比喻、擬人化或說故事的形式，讓沒有程式背景的人（甚至 5 歲小孩）也能秒懂該段代碼的作用。

*範例：*
```javascript
// [技術] 啟動背景間隔輪詢，定時平行發送請求更新所有監控中诊間進度
// [小朋友] 派出一組跑腿小弟，每隔一分鐘同時出發去醫院看板幫我們看診號碼
```

---

## 2. 🛜 資源管理與網路防漏 (Resource Management)

* **HTTP/HTTPS 請求釋放**：
  * 當使用 Node.js `https.request` 發送外部 Webhook 請求（如 Discord、Telegram）且**不需要讀取回傳資料**時，**必須**在回呼函式中調用 `res.resume()`。
  * *原因*：未消耗的回應串流會導致 underlying socket 被掛懸，長期運行下會造成記憶體洩漏與連線塞車。
* **SSE 連線計時器清理**：
  * 在 `/api/events` (Server-Sent Events) 端點中建立的任何計時器（例如 `setInterval` 的心跳包），**必須**在客戶端離線監聽器 `req.on('close', ...)` 中調用 `clearInterval` 進行銷毀，並將 response 物件移出連線池。
  * *原因*：若未清理，客戶端斷線重連後，伺服器會積累大量懸空計時器，導致 CPU 與記憶體耗盡。

---

## 3. 🛡️ 防呆與輸入驗證 (Robustness & Validation)

* **輸入值安全性審查**：
  * 凡涉及數值計算或狀態比較的欄位（如掛號號碼 `targetNumber`、提醒門檻 `alertThreshold`），在進行 `parseInt` 轉換後，**必須**檢查 `isNaN` 且驗證其是否為**正整數**（大於 0）。
  * *原因*：防止非法字串導致 `NaN` 帶入比較邏輯中，造成靜默失敗（Silent Failure）。
* **前端操作防呆 (Foolproof UI)**：
  * 診間輸入必須預設使用**動態叫號選單**（透過 `/api/clinics` 即時抓取高醫今日開放診間），且必須保有「手動輸入」作為備用 fallback 機制，確保在高醫網頁異常時系統仍可手動運作。

---

## 4. ⚙️ 環境與配置規範 (Environment & Port Config)

* **動態連接埠**：
  * 伺服器埠號（Port）必須從 `.env` 的 `PORT` 變數動態讀取，預設應為 **`5555`**（以防與本機其他開發服務如 nanoclaw 的 `3000` 埠衝突）。
* **測試代碼同步**：
  * 所有測試腳本（如 `test_server.js`）在呼叫 API 時，**必須**優先解析 `.env` 檔來獲取當前運行的 `PORT`，嚴禁硬編碼（Hardcode）地址，以確保測試工具隨時可用。
